## 象限扫描

### 已知的已知（今天要动手的）

- **LLM 主导的 ReAct 闭环**（教程 09 + 报告 §2.1）：Reason→Act→Observe，模型每步
  吐一个结构化动作 `{thought, tool, args}`，工具结果回灌为下一轮 observation。
  落点 `repair/agent.py:repair_finding`。**结构化 JSON ReAct 而非原生 function-
  calling**——M3 恒发 `<think>` 前缀、仓库已有硬化 JSON 解析（Day 5），复用这套
  纪律最省事（对应报告 Q1：JSON 不稳的根治是架构层，不是求模型"务必合法"）。
- **神经符号分工**（报告 §6 发展趋势 + 陷阱 1）：LLM 只输出**抽象意图**
  （"删掉这个 xpath 的节点"），底层由 lxml 做真正的 DOM 拼接。我们把它落成四个
  受约束的结构化 `EditOp`（set_attr/set_text/remove/insert，`repair/models.py`），
  **绝不给自由字符串/正则替换工具**（报告 §3.1 ACI 防呆：过度底层的工具会摧毁
  文件结构）。这直接消灭了"LLM 忘了闭合标签"这一类低级崩溃。
- **dry-run / apply 双模式**（报告 §3.2）：默认 dry-run，只在沙箱虚拟改、喂给真
  校验器、渲染 diff+理由；apply 需显式授权。落点 `repair/core.py` + CLI 默认无
  `--apply`。**这一层隔离是防止生成式 AI 污染生产核心数据的最后防线**。
- **闭环验证 = 唯一信任来源**（报告 §3.3 + 陷阱 4）：可信度不来自模型自称"修好
  了"，而来自**确定性校验器复跑**。落点 `tools.py:_propose_patch` → `validate()`
  调 Day 2 `analyze_package`。**验证器与生成器物理隔离**（报告 §5.4 共谋陷阱）：
  我们的 judge 是死板的 XSD/BREX 解析器，不是另一个 LLM——回音室从架构上被切断。
- **无限循环熔断**（报告 §5.1 + 陷阱）：三维预算断路器（迭代/token/no-progress），
  可配置（`[tool.learnarken.repair]`）。落点 `repair/config.py` + `agent.py`。
- **最小 diff / 爆炸半径约束**（报告 §5.2 + 陷阱 5）：patch 必须清掉目标 finding
  且**零新增**（over-repair 门），且 EditOp 单节点锚定天然限定改动范围。落点
  `tools.py:_propose_patch` 的 `Counter` 多重集判定。
- **Patch 带理由与引用**（报告 §3.4）：`ProposedPatch` 携带 rationale（ReAct
  thoughts 蒸馏）+ evidence（检索/读过的模块 id）+ validator_delta，落审计
  artifact `eval/repairs/`。死后剖析靠它。

### 已知的未知（动手前要定 / 要探测）

1. **apply 与宪法 §1.3 正面冲突**（本扫描的头号发现，已由 SPEC 前置问答裁）。
   §1.3"提议修复、待人批准，绝不静默改数据"是最高权威；报告 §3.2/Q5 也把 apply
   落盘描述为"必须人审+灰度"的最危险动作。**裁决 1：apply=批准后写入**——逐 patch
   人工闸，只自动化"批准→写盘"，不改宪法。这与报告 Q5"事务隔离+版本化+
   human-in-the-loop"完全同构。
2. **沙箱围栏做到多严**（报告 §3.1 ACI + §7 陷阱 3 故障隔离）。给 Agent 执行
   shell/Python 的能力是把双刃剑（报告开篇"释放一头野兽"）。**裁决 2：沙箱执行器**
   ——应用层围栏（AST import/属性白名单 + 临时目录 jail + rlimit + 超时），**诚实
   标 toy-scale（INV-7）：不是 OS 级隔离**。探测结论见"未知的已知"第 3 条——
   "allowed import 仍是能力"这个直觉盲点，是红队替我们咬到的。
3. **修哪些错**（报告 §4.2 失败分类里"不该修的诱饵"）。**裁决 3：全 L0–L3 尽力
   而为**，高风险类（L0 语法/L1 结构/VIO-6 跨域）强制 dry-run-only。VIO-6 的"修法"
   就是删整个模块——那是 §1.1 的人类范围决定，不是 Agent 能替代的。
4. **M3 当 ReAct 驱动未经检验**：它稳定吐 JSON，但多步工具选择质量未测。预算断路器
   是兜底，低默认 `max_iterations=12` 控成本。这条留作实跑观测（当前 CI 用脚本化
   LLM，hermetic）。
5. **目标绑定要在服务端**（报告 §4.2 定位失败 Grounding Failure）：模型可能找错
   节点/改错文件。动手时我把 target_key 放进了 render_user 让模型带回——**这是错的**
   （见红队 #4/#5），正确解是服务端绑定，不信任 LLM 的 args。

### 未知的已知（容易带错旧直觉的）

- **"给 Agent 自愈能力 = 给它一个 shell"**——错。报告 §3.1 的 ACI 防呆原则、
  §6"结构化数据修复是强约束受控环境"都指向反面：**高语义、强约束的结构化工具**，
  不是通用 shell。传统运维直觉（"给我 root 我自己修"）在这里是最危险的反模式。
- **"LLM 能自查它改得对不对"**（报告陷阱 4 + §5.4）——错，且是致命的假阳性来源。
  同族模型共享盲区 → 生成器写的洞，同源验证器也看不出。**唯一真理是确定性
  校验器**。我们从一开始就没让 LLM 当 judge，这条是做对了的。
- **"allowed import 是安全的"**——**本次最大的直觉盲点，是红队（#1）替我们咬到的**。
  我把 `pathlib`/`sys` 放进沙箱白名单、以为 AST 挡住 `open`/`socket` 就够了；实际
  `pathlib.Path.write_text` / `lxml.etree.parse(url)` 经**被允许的库**照样读写任意
  文件、抓 URL。教训：**在沙箱里，一个 import 就是一份能力**，白名单要按"这个库
  能做什么 I/O"审，不是按"这个库看起来人畜无害"。已修：去掉 pathlib/sys + 禁
  文件/网络方法属性。这正是报告没细讲、但工程上最容易翻车的地方。
- **"原子写 = os.replace 就完事"**——不够。红队 #10/#11 暴露：预览校验与换入之间有
  TOCTOU 窗口；恢复若信任任意 `.bak` 会被投毒。正确解是**预验证字节复检 + journal
  化恢复**（只认自己写的备份）。对应报告 Q5"不可篡改变更日志 + 反向恢复快照"。
- **"结构化数据修复容错高、随便改"**（报告 §6 对比表确实说 XML 回滚成本低）——
  别把"回滚便宜"读成"可以乱改"。VIO-4 改 DMC systemCode 会**波及所有 dmRef/DML/PM
  引用**（报告陷阱 2 拓扑依赖）；我们靠 over-repair 门（零新增）+ 人工闸兜住，但这
  提醒：结构化不等于无涟漪。

### 未知的未知（预埋观测点 / 红队补上的）

- **本扫描的盲点，跨宿主红队补齐了**：#2 shell 参数未 jail（`cat /etc/passwd`）、
  #3 源符号链接被拷进 jail、#5 打补丁文件与报告文件可背离——这三条我动手时都没
  想到，Codex 咬到了。**方法论收获**：非实现方模型的价值恰恰在实现方的思维盲区，
  这本身印证了报告 §5.4 的"异构验证"论点（换个模型族看同一段代码）。
- **多 finding 同文件的拓扑依赖排序**（报告陷阱 2）我们**没做**：当前逐 finding 独立
  修、apply 时把同文件的多个 EditOp 顺序叠加。若两个修复相互依赖（改父节点消掉子
  节点的伪报警），当前顺序可能不对。demo 语料小、暂未触发；留观测点，真出问题时
  引入"依赖拓扑图规划"。
- **投毒上下文**（报告 §5.3）：我们的 `search_corpus` 只检索**包自身**的 chunk 找
  兄弟模块的正确写法——语料就是被修的包，理论上存在"拿一个同样错的兄弟当范本"的
  风险。当前无置信度评估协议；留作未来（真接企业知识库时按报告 §5.3 加"规范×结构
  互验+主动拒修"）。
- **沙箱的 OS 级逃逸**（诚实 toy-scale）：应用层白名单挡不住恶意 C 扩展 /
  `__builtins__` 重构这类原生逃逸。宪法 §2 非敌意输入假设部分覆盖，但真上生产要
  换 nsjail/容器/seccomp。已在代码注释标"真 jail 该放哪"。
- **早期接受机制**（报告 §4.1 Early-Accept）我们隐式做了（propose_patch 一旦
  accepted 就 break），但没做"best-result 保留"（报告 §5.1）：熔断时返回的是
  REFUSED，不是"触发错误最少的中间版本"。留观测——多步修复变常见时值得加。

## 必须吃透的点（面试级）

1. **闭环验证是自愈系统的立身之本 + 生成器-验证器共谋**（报告 §3.3/§5.4，Q2）：
   可信度不来自模型自称"修好了"，来自**确定性校验器复跑**。要能说清共谋为什么致命
   （同族模型共享盲区 → 假阳性），以及我们的架构立场：judge 必须是死板的 XSD/BREX
   解析器，**绝不用 LLM 验 LLM**。这是"把保证放在结构里、不放在约定里"在 Day 7 的
   兑现（和 Day 5 引用确证、Day 6 哑终端同一立场）。
2. **神经符号分工消灭格式崩溃**（报告 §6/Q1，陷阱 1）：LLM 只输出扁平意图，lxml
   做 DOM 拼接。能具体说出我们的四个 `EditOp` + 单节点 xpath，比背"用 Structured
   Outputs"更有说服力。一句话立场：**模型负责'改哪、改成啥'，宿主代码负责'怎么改
   得语法正确'**。
3. **三维熔断防死循环**（报告 §5.1，Q3）："改 X→报错→改 Y→报错→改回 X"的震荡，本质
   是状态失忆 + 误导性报错。我们的三层防御：迭代上限、token 上限、**no-progress
   计数（连续被拒的 patch）**。要能接下半句：更进阶是 best-result 保留（熔断也返回
   离成功最近的中间版本），我们暂未做、已登记。
4. **最小 diff / 爆炸半径**（报告 §5.2/陷阱 5）：模型爱"表现欲"顺手重排整个文件。
   物理隔离网 = 落盘前算 diff，改动范围大幅偏离 finding 节点就拦截。我们用"单节点
   EditOp + 零新增校验"实现。能连到 VIO-4 改 DMC 的涟漪（陷阱 2 拓扑依赖）更佳。
5. **ACI 防呆 + "allowed import 就是能力"**（报告 §3.1 + 本项目红队 #1）：面向概率性
   LLM 的接口要防呆——不给底层自由工具，给强约束结构化工具。**最深的一课是沙箱：
   一个被允许的 import（pathlib/lxml）本身就是一份文件/网络能力**，白名单要按"能做
   什么 I/O"审。这是能讲出"我建了沙箱、红队证明它漏、我按'能力'而非'看起来安全'
   重审白名单"的完整故事——比只说"我加了沙箱"强得多。
6. **为什么"能用 Workflow 就别用 Agent"**（报告 Q4）：全自主 ReAct 循环自由度高，
   代价是不可预测的状态机、暴涨的 token、越权风险。对**目标明确、有确定验证标准**的
   校验修复任务，把控制权收回确定性控制流（LLM 只当特定节点的推理函数）更稳。我们
   的设计正是这个立场：ReAct 循环被夹在**确定性校验器 + 结构化工具 + 人工闸**之间，
   模型是受约束的推理节点，不是放养的自主体。**简单可控 > 智能本身**。
